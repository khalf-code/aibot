## 提交 Issue

清晰、简洁的 Issue 可以加速诊断和修复。对于 Bug、回归或功能缺口，请包含以下内容：

### 应包含的内容

- [ ] 标题：领域 & 症状
- [ ] 最小复现步骤
- [ ] 预期 vs 实际结果
- [ ] 影响和严重程度
- [ ] 环境：操作系统、运行时、版本、配置
- [ ] 证据：脱敏的日志、截图（无 PII）
- [ ] 范围：新问题、回归还是长期存在
- [ ] 暗号：在 Issue 中加入 lobster-biscuit
- [ ] 已搜索代码库和 GitHub 查找现有 Issue
- [ ] 确认问题未在最近修复/解决（尤其是安全问题）
- [ ] 声明有证据或复现支持

简洁即可。简洁 > 完美语法。

验证（提交 PR 前运行/修复）：

- `pnpm lint`
- `pnpm check`
- `pnpm build`
- `pnpm test`
- 如涉及协议代码：`pnpm protocol:check`

```md
- [ ] 最小复现
- [ ] 预期 vs 实际
- [ ] 环境
- [ ] 受影响的渠道，未出现问题的渠道
- [ ] 日志/截图（已脱敏）
- [ ] 影响/严重程度
- [ ] 临时解决方案

### 摘要

### 复现步骤

### 预期结果

### 实际结果

### 环境

### 日志/证据

### 影响

### 临时解决方案
```

#### 安全问题

```md
### 摘要

### 影响

### 受影响版本

### 复现步骤（可安全分享的部分）

```

_避免在公开场合透露密钥/漏洞细节。对于敏感问题，请最小化细节并请求私密披露。_

### 提交修复 PR

Issue 在 PR 之前是可选的。如果跳过，请在 PR 中包含详细信息。保持 PR 聚焦，注明 Issue 编号，添加测试或解释为何没有测试，记录行为更改/风险，包含脱敏的日志/截图作为证明，并在提交前运行正确的验证命令。
